# ADR-009: Deprecate the legacy `eberlight-explorer/` directory

## Context

The repository currently contains two parallel Streamlit applications:

- `explorer/` — the canonical app (`app.py`, 4 pages, 45 passing tests, lazy-load Interactive Lab, used by `scripts/build_static_site.py` for the Pages mirror).
- `eberlight-explorer/` — the **predecessor** referenced in ADR-001, ADR-002, and ADR-005 as "the existing explorer". It carries 6 YAML catalogs, a `build_index.py` step, an older Streamlit layout, and pre-redesign visuals.

The pre-redesign explorer was kept around during the v0.1 → v0.3 migration so design choices could be compared back-to-back. That migration is now closed: ADR-002 chose runtime note loading over a YAML index, ADR-005 replaced the visuals, and ADR-008 added the Interactive Lab — none of which exist in the legacy app.

Today the legacy directory is unreferenced from the production code path, untested in CI (`.github/workflows/test.yml` only installs `explorer/requirements.txt`), unmentioned in the new README, but it is still present and confusing for new contributors. README rewrites in PR #30 broke the contract that "this folder is for the old app and that one is for the new app" because the new README only mentions `explorer/`.

We need a clear, written-down outcome that is consistent across CLAUDE.md, README.md, the ADRs, and the file tree.

## Decision

**Mark `eberlight-explorer/` as deprecated and physically retain it on `main` only until `notes-v1.0.0` ships.** Concretely:

1. `eberlight-explorer/README.md` is rewritten to a single-page DEPRECATED notice that
   - links to `explorer/` as the canonical replacement,
   - links to ADR-001, ADR-002, ADR-005, ADR-008 explaining what changed,
   - states that the directory is read-only and **will be removed** at the `notes-v1.0.0` cut.
2. `eberlight-explorer/` is **excluded from CI** (already true) and from
   `scripts/build_static_site.py` (already true).
3. The legacy `requirements.txt` is **not** added to any workflow and is not
   maintained going forward; security updates land only in
   `explorer/requirements.txt`.
4. `README.md` (root) explicitly calls out the directory as legacy with a
   pointer to this ADR — done in PR #30.
5. CLAUDE.md does not list it under canonical content — done in PR #30.
6. At the `notes-v1.0.0` release, a follow-up PR removes the directory
   entirely. The `notes-v1.0.0` release notes will document the removal in
   their "Migration / Compatibility" section.

We do **not** delete it now because:

- ADR-001 / ADR-002 / ADR-005 use it as a reference point; their before-state
  argument depends on the artifact existing for at least one merged release.
- Some old git-blame trails resolve more cleanly if the file tree matches
  the ADR text it cites.
- 256 KB of dormant Python is a small carrying cost compared to the loss of
  context if it disappears mid-migration.

## Consequences

### Positive

- One canonical app per repo. New contributors see exactly one place to land.
- README, CLAUDE.md, ADRs, and the file tree finally agree.
- CI and the Pages mirror keep validating only the live code path; nothing
  silently rots in CI for the legacy directory.
- A clear cut-off (`notes-v1.0.0`) means we don't keep deferring the deletion
  forever.

### Negative / Trade-offs

- The repository carries 256 KB of dead Python through the v0.10.x → v1.0.0
  window. Acceptable.
- Anyone landing on `eberlight-explorer/` via an old bookmark will now see a
  DEPRECATED notice instead of a working app. Acceptable; the notice points
  to `explorer/`.
- If `notes-v1.0.0` slips, the deprecation drags on. Mitigation: explicit
  follow-up entry in this ADR.

### Follow-ups

- **`notes-v1.0.0` PR** must remove `eberlight-explorer/` and update CHANGELOG
  + release notes. ADR-009 is then marked `superseded` with a pointer to the
  removal commit.
- Any new ADR-001 / ADR-002 / ADR-005 successor that no longer needs the
  before-state reference can also trigger the cleanup earlier; in that case
  this ADR's status moves to `superseded` and the deletion happens in the
  same PR.

## Alternatives Considered

- **Delete now, in this PR.** Rejected for the reasons in "We do not delete
  it now" above — the existing ADRs cite the directory as their before-state.
- **Move to a `legacy/` subfolder.** Rejected as pure churn — git blame already
  tells the story without renaming.
- **Move to a separate `legacy/eberlight-explorer-v0.x` git tag and delete on
  `main`.** Considered. Cleaner long-term but adds an extra release artifact
  to track now. Re-evaluate at `notes-v1.0.0`.
